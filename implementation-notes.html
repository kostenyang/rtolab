rtolab — VCF 5.2.1 / 9.0 / 9.1 自動化部署實作筆記

Spec: 在 SELAB-Cluster (vCenter 7.0u3) 上 nested 部 VCF 5.2.1 + 9.0 + 9.1 三組環境並存. 每組 4 台 nested ESXi + 對應 VCF Installer / Cloud Builder, 自動 bring-up management domain.

Repo: github.com/kostenyang/rtolab

Design Decisions

Decision — 三組 VCF 並存的 IP / FQDN 分段

在 192.168.114.0/24 mgmt subnet 內切三段不重疊區段:

VersionVCF mgmtESXiOperations (9.0)VSP (9.1)
9.1.5 inst, .10 sddc, .11 vc, .12-.13 nsx.14-.17—.18-.29 pool, .43-.44 fqdn
9.0.34 inst, .35 sddc, .36 vc, .37-.38 nsx.30-.33.40-.42—
5.2.1.54 cb, .55 sddc, .56 vc, .57-.58 nsx.50-.53——

三組 FQDN 共存 (kosten-vcf{521,90,91}-*.rtolab.local), 寫進 kosten AD DNS, 不衝突.

Decision — Nested ESXi: clone-from-master 而非 OVA 直接部

原本想 William Lam 的 nested ESXi OVA 直接 import 但 vCenter 7.0u3 不收 vmx-20. 改:

  1. 每版本手動裝 1 台 master (DCUI install + ESXi installer ISO)
  2. SSH 跑 Configure-NestedEsxiSsh.ps1: 套 6 個 vSAN/LSOM workarounds + /Net/FollowHardwareMac=1 + auto-backup
  3. Export-NestedEsxiOva.ps1 把 master 匯出成 .ova golden image
  4. Deploy-FromGoldenOva.ps1 把 OVA 部 3 個 clone (esx02-04)
Decision — 5.2.1 跑 CB 5.2.1 GA 而非 5.2.3

原本拿到的 OVA 是 Cloud Builder 5.2.3.0-25219033, 但它寫死要 ESXi build 25205845 (8.0U3l/2026). 試 skipEsxBuildValidation CB 不認.

改抓 CB 5.2.1.0-24307856 GA, 要 ESXi build 24280767 (8.0U3b/2024-09), 可用 vcf-download-tool + esxcli software profile update -d <depot.zip> 升 (~5min/host).

Decision — 9.1 用 IIS 起本機 HTTP depot 給 installer

VCF 9.1 native 支援 HTTP no-auth offline depot (port 8888). PowerShell HttpListener 太慢 (504 timeout 在 multi-GB 檔案), 改裝 Windows Server 2022 內建的 IIS, MIME types 補齊 (.ova .tar .tgz .iso .sig .yaml .xml .zip), 對 VCF Installer 9.1 PUT /v1/system/settings/depot 回 DEPOT_CONNECTION_SUCCESSFUL.

Decision — 9.0 用 installer 自己內建的 online depot

9.0 installer 開了之後 user 自己用 web UI 配 token + 走 broadcom 線上 depot. 154 bundles 自動下載到 installer 本身, 我們不用搞 IIS.

Deviations from Spec

Deviation — Gateway 從 .1 改 .254

初版 inventory/lab.yaml 寫 network.mgmt.gateway: 192.168.114.1, 但 SELAB underlay 實際 gateway 是 .254 (跟 172.16.10.254 同 convention). .1 完全不存在, 從 clone 試 ping .254 reply 才確認.

導致前期 clone 部好之後從 jumpbox ping 不到 — return path 死在不存在的 gateway. 改完 inventory + Apply-CloneIp 重套全部 host, 立刻通.

Deviation — vmk0 force re-bind to vmnic0 MAC (不靠 /Net/FollowHardwareMac 自動)

從 master OVA clone 出來的 host, 因 state.tgz 內 vmk0 MAC 是 master 的, clone 新 vNIC MAC 收不到 reply. /Net/FollowHardwareMac=1 只在 boot 時 evaluate, post-boot 設沒用.

用 GuestOps API 跑 esxcli network ip interface remove --interface-name=vmk0; esxcli network ip interface add --interface-name=vmk0 --portgroup-name='Management Network' --mac-address=$(esxcli network nic list | awk '/vmnic0/{print $8}') 強制 re-bind. 烤進 master state.tgz 之後 clone 第一次 boot 就吃對.

Deviation — VCF 9.x SddcSpec schema 重寫 (5.2.x 假設不適用)

我原本的 9.1 bringup template 是參考 5.2.1 autodeploy 寫的, 但 9.x API 結構不同:

  • vsanSpec 從 top-level 改進 datastoreSpec.vsanSpec
  • dvsSpec.vmnics 改 dvsSpec.vmnicsToUplinks (array of {id, uplink} 物件)
  • vspClusterSpec 新加 (VCF Services Platform, 9.x 必填)
  • IpRange.start/end 改 startIpAddress/endIpAddress
  • networkSpecs.vlanId 要 integer (不是 string)
  • teamingPolicy 要小寫 (loadbalance_loadbased)
  • NSX TEP 用 ipAddressPoolSpec.subnets[].ipAddressPoolRanges
  • nsxTeamings + nsxtSwitchConfig.transportZones 要明確指定

schema 從 installer 內 /opt/vmware/vcf/sddc-manager-ui-app/.../assets/vcf-installer-openapi.json (354KB) 抽出來對齊.

Deviation — 9.0 spec version: 9.0.0.0 vs 9.0.2.0

9.0 VCF Installer 跑 9.0.2.0 (build 25151285), depot bundles 也是 9.0.2.0.* (SDDC Manager .25151285, ESX_HOST .25148076 等). spec 內 version 必須對齊 depot 有的版本 (= 9.0.2.0). 寫 9.0.0.0 會 FAILED_TO_VALIDATE_COMPONENT_FILE.

Tradeoffs

Tradeoff — secrets: sops vs 本機明文

原 spec 用 sops + age 加密. Windows 環境沒 sops binary 順手, age key 管理也是額外維護. 選擇: inventory/secrets/lab.yaml 明文 + 強 .gitignore (inventory/secrets/lab.yaml). 風險: 拿到 jumpbox = 拿到密碼; 但 lab 範圍可接受.

Tradeoff — ESXi 升級: 走 esxcli profile update vs ISO boot

5.2.1 ESXi 需從 build 24022510 升到 24280767. 兩種路徑:

  • (A) ISO boot + interactive DCUI Upgrade: 5 min/host, 但要人類在 console 按 F11. 4 台 = 20 min 純手動.
  • (B) esxcli software profile update -d <depot.zip>: 全自動 SSH + reboot. 但要 depot zip (不是 ISO).

選擇: (B) — user 提供 depot zip 後, scripts/Upgrade-NestedEsxi521-zip.ps1 把 zip SFTP 到每台 /vmfs/volumes/OSDATA-*, 跑 esxcli, 自動 reboot. ~10 min/host 但 0 互動.

Tradeoff — VCF Installer 9.x depot: 本機 IIS 還是 Linux nginx

William Lam 的 create_vcf9_depot_server_v3.sh 是 Linux nginx 路線. 我這邊 jumpbox 是 Windows.

  • Linux VM + nginx: 跟 Lam 的 script 完全 align, 但要部一台 VM, 多花 30 min
  • Windows IIS: 本機 jumpbox 直接裝, Install-WindowsFeature 5 min OK

選擇: Windows IIS. 已驗證 9.1 installer 透過它 PUT depot config 成功 + 抓 manifest + 抓 bundle 都通.

Tradeoff — VCF 9.1 vs 9.0 bringup 優先順序

9.1 schema 我們重寫好了, validation passes deployment spec + security, 但 depot bundle metadata 不完整 (download tool 只拉 binary, 沒拉 upgrade_info.xml) → bundles 還沒被 installer 認到.

9.0 installer 自己內建線上 depot fetch, 比較快, 154 bundles 直接到位.

選擇: 先衝 9.0 bringup (已 in-progress, deploy vCenter 中). 9.1 等 9.0 完成釋出時間再回頭.

Open Questions

Question — VLAN 115/116/117 (vmotion/vsan/overlay) 在 physical switch 通不通?

原始 spec 假設 outer vDS selab-dswitch 的 trunk portgroup pass VLAN 0-4094, MTU 9000. 實測 VLAN 114 (mgmt) 完全通; 115/116/117 vmkping MTU 9000 在 9.1 validation 是 WARNING (5.2.1 / 9.0 已 SUCCEEDED — 看來 5.2.1/9.0 對 jumbo 寬鬆些, 或實體 switch 之前剛好 trunk 過).

如果 physical switch 沒 trunk 115/116/117 + jumbo, bringup 後 vSAN/NSX 流量會跑不起來. 需要 user 端確認 / 補上 trunk + MTU 配置.

Question — 9.1 bringup 是否還要往下推?

9.0 已 bring-up, 已驗證流程通. 9.1 卡在 depot bundle metadata 結構問題, 補齊還要時間. 如果 9.0 跑通了相當於 VCF 9 線的 PoC, 是否還需要 9.1?

Question — 5.2.1 leftover state cleanup 自動化

5.2.1 第一次 validation 失敗 (build 不對) 之後, CB 已經先在 host 上創了 vcf-m02-vds01 standard vSwitch + vmk30 (NSX TEP). 升 build 後 validation 抓到 "vSwitch already exists / vmk30 exists" 需要手動清.

是否要做一個 Reset-NestedEsxi.ps1 把 host 還原到 vmk0/vSwitch0 only 的乾淨狀態, 給 retry 用?
